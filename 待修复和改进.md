## 未修复问题详细说明

### 2.6 — SQLite 连接不关闭

**位置**: `xueli/src/emoji/database.py:101-104`

**问题**: `_connection()` 方法每次调用 `sqlite3.connect()` 返回新连接，从不关闭。

```python
def _connection(self) -> sqlite3.Connection:
    conn = sqlite3.connect(self.db_path)
    conn.row_factory = sqlite3.Row
    return conn
```

**修复方向**: 改用 `aiosqlite` 或通过上下文管理器显式 close。

---

### 3.2 — qsize() 无锁读取

**位置**: `xueli/src/core/model_invocation_router.py:220`

**问题**: `pending = self._queues.get(purpose_key).qsize()` 在锁外读取，值可能不精确。

**修复方向**: 日志可接受不精确值，或持锁读取（需评估死锁风险）。

### 3.10 — 双配置系统

**问题**: `start.sh` 检查 `xueli/config/.env`，但 Python 代码只用 `config.toml`。`.env` 文件对 Python 运行时无效。

**修复方向**: 统一为单一配置来源（推荐 `config.toml`），删除 `.env` 相关逻辑。

### 4.2 — Lock 在 async 方法内创建

**位置**: `xueli/src/handlers/conversation_plan_coordinator.py:831`

```python
conversation_lock = self.conversation_plan_locks.setdefault(group_id, asyncio.Lock())
```

**修复方向**: 延迟创建锁到需要时，或预先填充 `conversation_plan_locks`。

## 待改进 — Timing 决策后续行为

**问题描述**: 当 Timing Gate 决策为 `wait` 时，当前行为是"标记当前窗口完成，尝试取出队列中的下一条消息"。这个行为对于私聊场景可能过于激进。

**当前行为**:
```
timing_decision = "wait"
  → _build_dispatch_plan_from_timing(timing_decision)
  → action = WAIT, source = "timing"
  → _try_dispatch_next_window(plan)
  → complete_window_dispatch(plan) → mark_window_complete
  → 尝试取出队列中的下一条消息处理
  └── 有下一条 → 继续主循环
  └── 无下一条 → 退出，消息处理结束
```

**改进方向**:
- 私聊场景：Timing Gate 为 `wait` 时，应该保留当前消息在窗口中等待更多输入，而不是立即分发给下一条
- 群聊场景：可以维持当前行为（因为需要判断是否插话）
- 或考虑：当 `wait` 时不立即 complete_window，而是等待静默期后再判断

**相关文件**: `xueli/src/core/runtime.py`
